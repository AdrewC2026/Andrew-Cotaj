§ 01 / The idea

A matching engine is a pure state-transition function.

This is the exchange side, not a trading bot: the engine receives order commands, maintains the limit order book, matches crossing orders by price-time priority, and emits executions. Everything is organised around one pure function: 

apply : book -> command -> book * events

Purity gives deterministic replay for free; totality is enforced by exhaustive pattern matching over every command and order type; invariant preservation is what the property tests check after every application. OxCaml — Jane Street's production compiler fork of OCaml — then makes the same logic fast and concurrency-safe through modes, without dropping into C.

The engine is built vanilla-first: a correct, idiomatic plain-OCaml version is tagged (v1-vanilla) before any OxCaml extension is added, so the v1-vanilla → HEAD diff is the evidence of the mode work, and every performance claim is a measured number rather than an assertion.

§ 02 / Type safety

Illegal states don't typecheck.

1 · Abstract types + validating smart constructors

A Price.t is provably ≥ 0 and a Qty.t provably > 0, because those are the only constructors and they validate. No module downstream ever needs a defensive check.

module Price : sig
  type t : immutable_data          (* abstract — no raw constructor escapes *)
  val of_int : int -> t option    (* Some iff n >= 0 *)
end

2 · A GADT order lifecycle

The two states carry different payloads — Filled has no remaining field at all:

type _ t =
  | Active : { id; side; price; remaining : Qty.t; seq : int } -> active t
  | Filled : { id; side; price;                    seq : int } -> filled t

val remaining : active t -> Qty.t     (* a filled order has no remaining to read *)
val fill : active t -> Qty.t -> [ `Partial of active t | `Filled of filled t ]

Forbids — reading the remaining quantity of a filled order, and re-filling a filled order. Both are compile errors, not runtime checks: fill consumes the active order and returns a fresh value.

3 · Polymorphic-variant time-in-force rows

A market order's TIF row omits `Gtc / `Post_only, and Market has no price field:

type kind =
  | Limit  of { price : Price.t; tif : [ `Gtc | `Ioc | `Fok | `Post_only ] }
  | Market of {                  tif : [ `Ioc | `Fok ] }   (* no price, no Gtc *)

Forbids — a resting market order and a post-only market order are unrepresentable: you cannot pass `Gtc where a market TIF is expected, and a market order has no price to rest at.

§ 03 / The engine

Book, matching, and the invariants.

The book is two price-keyed Maps (O(log n) best-price access), each level a FIFO functional queue; time priority is structural — a monotonic sequence number plus append-at-back. Matching walks the opposite side best-price-first and handles limit, market, IOC, FOK (with a pre-check pass), and post-only, executing at the resting (maker) price. Because apply is pure, a text command stream replays to byte-identical output.

Correctness is a qcheck property suite that asserts five invariants after every step across thousands of randomised command streams:

InvariantStatement checked after every command
Share conservationsubmitted = resting + cancelled + 2 × filled
No crossed bookbest_bid < best_ask whenever both exist
FIFO within a levelsequence numbers strictly increase front-to-back
Price-time prioritya taker's fills walk prices best-first
Deterministic replaythe same stream twice → identical book & events

Result: 7 / 7 properties pass (5 invariants + 2 smart-constructor round-trips, 2000 streams each) and 16 / 16 hand-written regressions pass — on both the vanilla and OxCaml switches.

§ 04 / OxCaml · core 1

A zero-allocation hot path.

In vanilla OCaml, every intermediate fill event, updated order record, and list cons in the matching loop allocates and feeds the GC. OxCaml's local mode keeps that data on the stack. Over a flat, cache-friendly price level, the only thing that could allocate is the per-fill datum — and @ local / stack_ put it on the stack: 

val match_sweep_local :
  level -> price:int -> taker_qty:int -> on_fill:(fill @ local -> unit) -> int
(* each fill is `stack_`-allocated and handed to on_fill as `@ local`, so it
   cannot escape onto the heap. Drop the `@ local` and the records heap-allocate. *)

Identical work (checksums match); only the annotation differs. Zero allocation ⇒ zero minor collections ⇒ no GC tail on the hottest path in the system.

§ 05 / OxCaml · core 2

A data race is a compile error.

A multi-domain order-entry gateway feeds a single matching domain. OxCaml's portability / contention modes make "no data race on the book" a compile-time guarantee. Commands cross domain boundaries as @ portable values with no cast, because their leaf types carry : immutable_data kinds; the mutable book is created inside the matching domain and never crosses a boundary: 

type t : value mod portable contended        (* the cross-domain hand-off *)
val submit : t -> command @ portable -> unit @@ portable   (* any domain *)
val run    : t -> int Domain.t                          (* the one writer *)

The proof is a rejected program — sharing one mutable Book.t ref across two domains:

Error: The value "book" is "nonportable" but is expected to be "portable"
       because it is used inside the function ... which is expected to be "portable".

Measured throughput of the accepted gateway: ~2.6–2.9 M orders / sec (4 producer domains → 1 matching domain). "This does not compile, here is the error" is the strongest evidence for data-race-freedom by construction.

§ 07 / Honest engineering

What's real, and what's a trade-off.

The four casts. The gateway uses exactly four %identity mode casts, audited individually, only for facts the deliberately-un-kinded engine hides: that Book.t is deeply immutable (the empty seed) and that Matching.apply / Book.total_shares are pure. The concurrency mechanism itself — the portable closure, the command hand-off, the shared atomics — uses no casts.

Scoped to two extensions. Allocation control and mode-based race freedom, understood completely, over feature breadth. Unboxed int64# layouts were not pursued — an extension without a measured number is left out rather than decorated in.

HEAD targets OxCaml. The : immutable_data kinds live in the engine, so HEAD requires the 5.2.0+ox switch; the standalone plain-OCaml guarantee is preserved at the v1-vanilla tag. Every annotation in the repo is paired with a "why it typechecks / what escapes if you remove it" comment.
